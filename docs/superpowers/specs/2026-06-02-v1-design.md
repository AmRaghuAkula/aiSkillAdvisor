## 1. What this is (naming + identity)

**aiSkillAdvisor is a Claude Code _plugin_** that acts as an always-on **advisor** ("a co-pilot for your co-pilot"). It watches your work and proactively surfaces the right skill at the right moment — aimed at **non-technical builders** (founders, PMs, designers, indie hackers) who have product instincts but miss skill opportunities.

The naming question is resolved by recognizing these are **different layers, not competing names**:

| Layer | Term | Meaning here |
|---|---|---|
| What a user installs | **Plugin** | The one-command, distributable package. The distribution vessel. |
| How it behaves inside | a **Skill** (the routing brain) + **Hooks** (the always-on listener) | The mechanism. |
| What we call it to humans | an **Advisor** | The product category / marketing name. |

It is **not** an "agent" (agents are dispatched for one bounded task then end; the advisor lives alongside the whole session continuously). It **must** be a plugin (not a bare skill) because only a plugin can run hooks (continuous monitoring), bundle skill + hooks together, install via one command, and install other skills on the user's behalf.

**Platform:** **Claude-first for v1**, with a **portable brain**. The advising logic (work-type classification, routing, loop-prevention, value log) is designed to be platform-agnostic; only the thin integration layer (hooks, plugin packaging, install motion) is Claude-specific. Multi-platform (Cursor, Aider, Azure AI Foundry, etc.) is a later version — deferred because the core value (routing among installed skills) only exists where a rich skill ecosystem exists, which today is Claude Code.

## 2. Core principle: inventory-relative

The advisor **never reasons against a fixed catalog.** "180+ skills" describes one maintainer's machine; another user (e.g., a PM on basic Claude) may have 30, 5, or none of the same skills. The advisor's behavior is therefore a **function of discovered state**, split into two jobs:

- **Job A — route among what you have:** suggest installed skills at the right moment. *(v1 core.)*
- **Job B — recommend what you don't have but should:** sweep trusted sources, surface a shortlist, help you add them. *(v1 included, with install kept opt-in — see §5.)*

When no installed skill fits a moment, the advisor says so in plain English rather than going silent ("No installed skill for this yet — you could add one; here's a match from a trusted source").

## 3. The runtime mechanism (decision: "automatic tap + portable brain")

Today's v0 lives as a memory file whose `description:` field makes Claude *recall* it when relevant. That is recall-based ("speaks up when it remembers") and is exactly why v0 sometimes goes quiet (documented in the v0 TDD log: a missed `brainstorming` fire). v1 fixes this with a **deterministic tap**:

- **The tap (hooks):** the *harness* runs it — not the model — so it cannot be forgotten. Fires at:
  - **session start** — load the user profile, run the local inventory sweep, pin the goal anchor (L1).
  - **each user prompt / before Claude acts** — classify the work-type of the turn and feed the brain.
- **The brain (skill):** holds the actual judgment — the existing v0 algorithm carried over wholesale (work-type classification, routing CASES A–F, Loop-Prevention L1–L5, red-flag tripwires, open-world rule, plain-language output). Portable.

**Why this and not the alternatives:** "speaks up when it remembers" (recall-only) is the v0 status quo and demonstrably drifts. "Automatic tap only" (all logic in scripts) is reliable but shallow. The tap-plus-brain split is reliable *and* smart, and keeps the smart half portable for future platforms.

## 4. The five components

1. **The tap (always-on listener)** — hooks that fire deterministically at session start and on every prompt/response. *(New; the piece v0 lacks.)*
2. **The brain (advisor logic)** — the v0 algorithm, carried over. Reads each prompt+response, names the work-type(s), picks the right skill(s). Portable.
3. **Inventory awareness** — at session start, reads what skills are actually installed on this machine (the *local* sweep). Routes only among those; speaks plainly when nothing fits.
4. **Who-you-are profile (light onboarding)** — pick a persona (founder / designer-who-codes / PM-in-tech / indie hacker / hobbyist), tweak it, or add your own → a plain-English profile the brain consults to weight suggestions. Non-blocking (running in ~10s); partial completion still yields a usable profile.
5. **Value memory** — a local-first event log records every advisor action (suggested / accepted / declined) and especially **near-misses caught** (a risky action prevented — e.g., a billing change about to ship with no security review). Produces a plain-language end-of-session / daily summary of quality gained + mistakes prevented. **The log is written from day one even though the polished report UI comes later** (you cannot report value you never logged).

## 5. Onboarding + discovery flow

Light onboarding, non-blocking, value visible early:

1. **Pick a persona** (~10s). Editable line-by-line; "this isn't me" escape always available; can add a custom persona.
2. **"Want me to look at the skills you already have?"** → **local system sweep** → shows them in plain English. *One click, runs automatically (read-only, safe).*
3. **"Want me to check trusted sources for skills worth adding?"** → **trusted-repo sweep** → presents a **persona-filtered shortlist of 5–7** (never the whole catalog — avoids overwhelm).
4. **For any the user picks → "Install it for you?"** → **opt-in / one-tap install** (see install rule below).

**Design decisions baked in:**

- **Order:** system sweep *first*, trusted-repo sweep *second* — so the advisor never recommends a skill the user already has.
- **Trusted sources only (v1):** a curated, vetted registry ships with the plugin (seed: `anthropics/knowledge-work-plugins`, gstack, vercel, superpowers ecosystems). Paste-any-GitHub-URL is **deferred** (too easy for a non-technical user to add something unsafe).
- **Reusable, built once:** the sweep is the same action later exposed as a "refresh skills" button — available anytime, not just at onboarding.
- **Progressive disclosure:** surface ≤5–7 skills at any moment; outcome-phrased labels (no skill jargon in primary text; jargon only in tooltips).

## 6. The one safety rule: read = automatic, change = ask

Carried over and extended from v0 (lines 149–151 of the live advisor):

- **Read-only / advisory skills** (`brainstorming`, `spec`, `writing-plans`, `code-review`, `design-review`, `verification-before-completion`, `investigate`, `browse`, `qa-only`, `health`) → **auto-run, announced** ("running a quick design review — back in ~30s"). No approval gate.
- **State-changing actions** (`ship`, `deploy`, migrations, **and installing a skill**) → **always ask first.** One tap to approve.

**Install (v1 = opt-in / manual, friendly version):** the advisor does *all* the prep, then proposes the install as a **one-tap "approve to proceed."** On approval it performs the install, tells the user in plain language *"Installed X — restart Claude Code once to switch it on,"* and **verifies after restart** that the skill actually resolves. It never writes to the user's setup without that tap. *Auto-install (no tap, fully hands-off) graduates to v1.1 once the motion is proven.*

**Hard constraint (Claude Code reality):** new skills are discovered only at harness startup — there is **no mid-session hot-load.** The install flow MUST (a) complete the full motion incl. the `enabledPlugins` flag in settings, (b) tell the user a restart is required in plain language (never report bare "installed"), (c) verify post-restart and surface a clear failure if the skill doesn't resolve. *(A silently-ignored install is worse than no install — this is the documented F11 silent-failure bug.)*

## 7. v1 scope (explicit in/out)

**In v1:**
- The always-on tap (hooks) + the portable brain (v0 algorithm carried over)
- Local inventory sweep (what you have)
- Trusted-repo sweep (what you could add) — curated sources only
- Opt-in / one-tap install with restart-aware, verified motion
- Light onboarding (persona pick + tweak + custom) → plain-English profile
- Value event log (written from day one); basic end-of-session summary

**Out of v1 (deferred):**
- Fully hands-off auto-install (→ v1.1)
- Paste-any-GitHub-URL discovery (→ later)
- Full onboarding wizard with live preview / show-don't-ask task gallery (→ later)
- Polished value-reporting dashboard (log now, report UI later)
- Multi-platform (Cursor / Aider / Foundry) (→ v2)
